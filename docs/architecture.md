# Core Architecture (v1.0 Phase One)

## Overview of Core Modules

MintCore v1.0 Phase One is organised into focused, single-responsibility modules:

```
src/
├── core/              MintEngine, BatchMintEngine, TransactionBuilder
├── adapters/          LibauthAdapter — bridges engine and @bitauth/libauth
├── cashTokens/        CashTokens helpers: baton, category, mint, BCMR generator
├── scripts/           Locking-bytecode builders: P2PKH, P2SH, OP_RETURN, burn
├── transactions/      Transaction builders: mint, baton, send
├── types/             Shared TypeScript type definitions
│   ├── TokenPrimitives.ts   Branded scalars and token interfaces
│   ├── UtxoTypes.ts         UTXO models
│   ├── MetadataSchema.ts    Metadata schema v1
│   ├── TokenSchema.ts       Mint request schema
│   ├── MintConfig.ts        SDK configuration
│   └── ...
└── utils/             Utilities: hex, keys, fee estimation, validation
```

### `src/core/`

The engine layer. `MintEngine` is the single-token orchestrator; it validates the `MintConfig`, resolves UTXOs via the configured provider, and delegates transaction construction to `TransactionBuilder`. `BatchMintEngine` extends this with planning and multi-transaction execution. `TransactionBuilder` assembles, signs, and serialises CashTokens genesis transactions.

### `src/adapters/`

`LibauthAdapter` encapsulates all direct calls to `@bitauth/libauth`, keeping the rest of the codebase free of low-level cryptographic primitives and serialisation details.

### `src/cashTokens/`

Higher-level CashTokens helpers: baton management, category derivation, single-token and batch mint helpers, and `generateBcmr` / `hashBcmr` for BCMR document generation.

### `src/scripts/`

Stateless bytecode builders for composing BCH locking scripts (`p2pkhLockingBytecode`, `opReturnLockingBytecode`, `burnLockingBytecode`, and P2SH helpers).

### `src/transactions/`

Composed transaction builders for mint, baton-transfer, and send operations. These sit above `TransactionBuilder` and implement specific transaction patterns.

### `src/types/`

All shared TypeScript interfaces and branded scalar types. This module has zero runtime dependencies and defines the data contracts consumed by every other module.

### `src/utils/`

Pure utility functions: hex encoding/decoding, private-key validation, fee estimation constants and helpers, UTXO coin-selection, and schema validation.

## UTXO Model

MintCore uses a two-tier UTXO model defined in `src/types/UtxoTypes.ts`:

### `BaseUtxo`

The minimal BCH UTXO representation:

| Field | Type | Description |
|-------|------|-------------|
| `txid` | `string` | 64-char lowercase hex transaction id (display byte order) |
| `vout` | `number` | Output index within the transaction |
| `satoshis` | `number` | Value in satoshis |
| `lockingScript` | `string` | Even-length lowercase hex locking bytecode |

### `TokenUtxo`

Extends `BaseUtxo` with a `token` field carrying CashTokens data. A token UTXO may hold fungible tokens, an NFT, or both simultaneously.

| Field | Type | Description |
|-------|------|-------------|
| `token.category` | `TokenCategory` | Genesis txid identifying the token class |
| `token.amount` | `TokenAmount?` | Fungible token balance (omitted for pure NFT outputs) |
| `token.nft` | `object?` | NFT capability and commitment (omitted for pure FT outputs) |

Provider logic (`src/core/providerUtils.ts`) fetches `BaseUtxo` arrays from Chronik or ElectrumX and maps them to `TokenUtxo` where token data is present. Consumers of the SDK interact only with these typed interfaces.

## Metadata Schema

Metadata Schema v1 (`src/types/MetadataSchema.ts`) is a lightweight, versioned structure for describing CashTokens-based assets:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | `number` | ✓ | Must be a positive integer; always `1` for v1 documents |
| `name` | `string` | ✓ | Human-readable asset name |
| `description` | `string` | — | Short description of the asset |
| `attributes` | `Record<string, unknown>` | — | Arbitrary key-value pairs for extensibility |

The schema is intentionally minimal. Extended metadata (icons, URIs, localised names) belongs in a [CHIP-BCMR v2](https://github.com/bitjson/chip-bcmr) document generated by `generateBcmr()` and referenced via `TokenSchema.bcmrUri`.

## What Is Intentionally Not Included

Phase One establishes the minimal SDK surface needed to build, sign, and broadcast CashTokens genesis transactions. The following are explicitly out of scope:

- **Wallet connectivity** — no WalletConnect, browser extension integration, or hardware wallet support. Signing requires a raw 32-byte private key.
- **UI components** — no React, Vue, or any frontend framework code.
- **Bundled network client** — consumers supply a Chronik or ElectrumX URL; the SDK does not embed one.
- **Key management** — beyond deriving a P2PKH address from a private key, no key-generation, HD derivation, or key-storage logic is included.
- **Game mechanics / RNG / pack logic** — no random number generation, loot-box patterns, or game-specific minting rules.
- **Covenant templates** — no CashScript or comparable contract-level abstractions.
- **Advanced querying** — on-chain data access is limited to UTXO fetching required for fee funding; no indexer queries, token history, or balance aggregation.

All advanced features must live in separate modules built on top of the SDK.
